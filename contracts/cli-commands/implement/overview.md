# `implement` Command Contract — Overview

## Purpose
Execute a plan file end-to-end: implement each open task with a worker AI, validate the result through build, test, and adversarial-review gates, and mark each accepted task as complete in the plan file.

## Invocation
    npx flanders implement [plan]

- When `[plan]` is given, it is treated as the path to a plan markdown file under the `plans/` folder.
- When `[plan]` is omitted:
  - If `plans/` contains exactly one file, that file is selected automatically.
  - If `plans/` contains more than one file, the command exits non-zero with a diagnostic that lists the available plan files and instructs the user to re-run with the `[plan]` argument naming the one to implement.
  - If `plans/` contains no files, the command exits with an error.
- The project must be a git repository; the command requires git and has no flag to disable it. See `cli-commands/implement/git-integration.md`.

## High-level flow
1. Load the Flanders configuration from `.flanders/` per `shared/flanders-config.md`. The loaded configuration determines the AI tools and models used for the worker and reviewer stages.
2. Read the selected plan file.
3. Validate the plan file against `shared/plan-file-format.md` programmatically before doing anything else, and verify at least one task line was detected. On failure, exit non-zero with a diagnostic that names the problem, the path to the offending file, and the offending line. This validation runs unconditionally because plan files generated by `/flanders-plan` are no longer validated at generation time.
4. Detect which tasks are open and which are completed using the rules from `shared/plan-file-format.md`.
5. If every task is already marked complete, print `tasks completed` and exit successfully.
6. Run the git preflight check (see `cli-commands/implement/git-integration.md`), which requires the project to be a git repository and the working tree to be clean except for the selected plan file. If the preflight fails, exit non-zero before setting up any workspace.
7. Set up the run workspace (see `cli-commands/implement/workspace.md`).
8. Run the iteration loop (see `cli-commands/implement/iteration-loop.md`), advancing one task at a time.
9. Throughout the run, render the live UI defined in `cli-commands/implement/ui.md`.

## Outcomes
- Missing configuration — exit non-zero at startup when neither a project-scope nor a global-scope `.flanders/` is found. The diagnostic points the user at `npx flanders install`.
- Ambiguous plan selection — exit non-zero at startup when `[plan]` is omitted and `plans/` contains more than one file. The diagnostic lists the available plan files and instructs the user to re-run naming one.
- Plan validation failure — exit non-zero at startup when the plan file is missing, empty, or contains malformed task lines. The diagnostic names the problem and the path to the file.
- Git preflight failure — exit non-zero at startup when the project is not a git repository, or when the working tree has uncommitted changes other than the selected plan file. See `cli-commands/implement/git-integration.md`.
- `tasks completed` — printed and exit successfully when every task in the plan was already marked complete at startup.
- `all tasks completed` — printed and exit successfully when every remaining task was implemented and accepted during this run.
- Hard stop — printed and exit non-zero on a hard stop: when the per-task iteration cap is exceeded for any single task, or when the prep optimization runs and fails irrecoverably (see `rules/ai/task-context/prep-optimization.md`).
