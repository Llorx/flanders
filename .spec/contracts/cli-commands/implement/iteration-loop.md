# `implement` Command Contract — Iteration Loop

## Purpose
Define the loop that picks the next open task, drives a worker AI through implementation, validates the result through build, test, and adversarial-review gates, and either marks the task complete or briefs the next iteration about what failed.

## Per-run state
- `iteration` — counter for the current task. Reset to 0 every time a new task is picked.
- `MAX_ITER` — fixed upper bound of 5 iterations per task. Hardcoded; not configurable.

## Outer loop
While there is an unchecked task in the plan file:
1. Pick the next unchecked task. Capture its line number in the plan file, its title verbatim, and its full task text — the verbatim region running from its task line through its body, delimited per [.spec/contracts/shared/plan-file-format.md](/.spec/contracts/shared/plan-file-format.md) — all with no summarization or rewriting.
2. Reset `iteration` to 0.
3. Run the inner loop (below).
4. On a successful inner loop, the task's checkbox is flipped from open to done in the plan file. Continue with the next task.

When no unchecked tasks remain, print the all-tasks-completed message — one variant chosen at random from its pool (see [.spec/contracts/cli-commands/implement/overview.md](/.spec/contracts/cli-commands/implement/overview.md)) — and exit successfully.

## Inner loop (per task)
Each iteration walks through the stages below in order. Any stage that fails writes context to the `error.log` file inside the temporary folder and restarts the inner loop at stage 1; the next iteration's worker prompt automatically includes the previous-iteration briefing because the iteration counter is greater than 1.

1. Increment `iteration`. If `iteration` exceeds `MAX_ITER`, hard stop: print an error that names the task and points at the workspace logs, and exit non-zero.

2. **Worker stage.** Spawn a worker AI (via the AI runner — see [src/ai/.spec/contracts/ai-runner.md](/src/ai/.spec/contracts/ai-runner.md)). The AI tool and model used for the worker are the ones selected in the Flanders configuration (see [.spec/contracts/shared/flanders-config.md](/.spec/contracts/shared/flanders-config.md)).

   How the worker is launched depends on the iteration, per [src/commands/.spec/rules/ai/task-context.md](/src/commands/.spec/rules/ai/task-context.md):
   - **Iteration 1.** A fresh worker invocation. The orchestrator injects the full task text into the prompt and consolidates the content of every contract and rule the task references into a `spec.md` file in the worker's temporary folder that the prompt directs the worker to read in full, so the worker has the task and its obligations without locating and opening each file (see [src/commands/.spec/rules/ai/task-context.md#the-workers-first-iteration-receives-the-task-and-reference-content-by-deterministic-script-injection](/src/commands/.spec/rules/ai/task-context.md#the-workers-first-iteration-receives-the-task-and-reference-content-by-deterministic-script-injection)).
   - **Iteration n>1.** The worker continues from the work it produced in iteration 1 by session resume, so prior tool calls and prior reasoning made by this task's worker remain available; the task text and the referenced content are not re-injected (see [src/commands/.spec/rules/ai/task-context.md#the-worker-resumes-its-captured-session_id-across-iterations-of-the-same-task](/src/commands/.spec/rules/ai/task-context.md#the-worker-resumes-its-captured-session_id-across-iterations-of-the-same-task)).

   The worker prompt contains:
   - The plan file path.
   - On iteration 1, the full task text — the task's line plus its body, injected into the prompt verbatim per [.spec/contracts/shared/plan-file-format.md](/.spec/contracts/shared/plan-file-format.md) — and a directive to read, in full, the `spec.md` file the orchestrator consolidated in the worker's temporary folder, which holds the content of every contract and rule the task references (the linked section when the reference carries a section anchor, the whole file when it does not). On iterations n>1 these are not repeated.
   - Instructions to respect the obligations of every contract and rule the task references, implement the task, and update or extend tests so the new behavior is covered.
   - The full list of contract files and the full list of rule files (both by relative path from the project root, where each path is the contract's or rule's namespace). The worker may consult any file in these lists at its discretion in addition to the ones the task references and whose content was injected.
   - Instructions stating that, if the implementation changes how the project builds or how its tests run, the worker also updates the build and test scripts inside the temporary folder.
   - On every iteration after the first, the previous-iteration briefing (see below).

   When the worker invocation completes, and before the build gate runs, the orchestrator stages every change in the working tree with `git add -A`, so the worker's output — including any files it created that were never tracked — is in the index for the gates that follow and for the eventual commit (see [.spec/contracts/cli-commands/implement/git-integration.md](/.spec/contracts/cli-commands/implement/git-integration.md)).

3. **Build stage.** Run the build gate — the build script (when one exists) — per the gate semantics of [.spec/contracts/shared/build-test-validation.md](/.spec/contracts/shared/build-test-validation.md). On a failing gate, capture both stdout and stderr to the `error.log` file inside the temporary folder and restart the inner loop.

4. **Test stage.** Run the test gate — the test script (when one exists) — per the gate semantics of [.spec/contracts/shared/build-test-validation.md](/.spec/contracts/shared/build-test-validation.md). On a failing gate, capture both stdout and stderr to the `error.log` file and restart the inner loop.

5. **Adversarial review stage.** The Flanders configuration holds an ordered list of one or more reviewers (see [.spec/contracts/shared/flanders-config.md](/.spec/contracts/shared/flanders-config.md)), each with its own AI tool, model, and effort that may be the same as or different from the worker's and from one another's. The orchestrator launches the configured reviewers for this iteration — subject to the weighted-review configuration below, under which an optional reviewer may be cancelled once the round can complete without it — and only then forms a single verdict from the combined output of the reviewers that ran to a verdict.

   Each reviewer has its own temporary folder — created at run start alongside the main temporary folder per [.spec/contracts/cli-commands/implement/workspace.md](/.spec/contracts/cli-commands/implement/workspace.md), independent of the main folder and of every other reviewer's folder — and that reviewer's verdict file is the `error.log` inside it. Before launching the reviewers, the orchestrator deletes every per-reviewer `error.log`, leaving the per-reviewer folders themselves in place, so none of those files exist when the reviewers start. The orchestrator then launches all reviewers concurrently — they are read-only on the project, so running them in parallel is safe — each as its own AI-runner invocation. Every reviewer invocation for the task is a fresh invocation with no continuity from any prior reviewer invocation and no continuity between the reviewers of the same iteration. Each reviewer's own retries and rate-limit waits are absorbed independently by its runner invocation (see [src/ai/.spec/contracts/ai-runner.md](/src/ai/.spec/contracts/ai-runner.md)). The stage's reviewer work is complete once the round-completion condition below is met.

   The reviewers carry the weighted-review configuration from `.flanders/config.json` (see [.spec/contracts/shared/flanders-config.md](/.spec/contracts/shared/flanders-config.md)): each reviewer is either required or optional, and the configuration names the minimum number of reviewers that must run to a verdict in this round. The orchestrator does not decide reviewer by reviewer whether to abandon a usage-limited reviewer. Instead, on every reviewer transition — a reviewer finishing with a verdict, or a reviewer entering a usage-limit wait — it re-evaluates whether the round may complete. The round may complete once no reviewer is still running, every required reviewer has produced a verdict, and at least the configured minimum number of reviewers have produced a verdict. Until then the round continues, so a reviewer sitting in a usage-limit wait is left to run: its usage limit may clear and it may still produce a verdict — for example while a required reviewer counts down a multi-hour usage-limit wait. At the instant the round may complete, the orchestrator cancels every reviewer still in a usage-limit wait — necessarily an optional one, since every required reviewer already has a verdict — and forms the verdict from the reviewers that ran to a verdict. A cancelled reviewer produces no per-reviewer error file, is never re-launched, and does not contribute to the stage verdict. A reviewer that fails with an error rather than a usage-limit wait is never part of this decision: whether or not it is optional, its error follows the ordinary reviewer-error path so its message reaches the worker. See [src/commands/.spec/rules/ai/agents.md#a-review-round-completes--cancelling-any-still-waiting-reviewers--once-no-reviewer-is-running-every-required-reviewer-has-a-verdict-and-the-minimum-is-met](/src/commands/.spec/rules/ai/agents.md#a-review-round-completes--cancelling-any-still-waiting-reviewers--once-no-reviewer-is-running-every-required-reviewer-has-a-verdict-and-the-minimum-is-met).

   Each reviewer is launched fresh and receives the same deterministic provisioning the worker's iteration 1 receives (see [src/commands/.spec/rules/ai/task-context.md#every-reviewer-invocation-is-fresh-and-receives-the-deterministic-script-injection](/src/commands/.spec/rules/ai/task-context.md#every-reviewer-invocation-is-fresh-and-receives-the-deterministic-script-injection)): the full task text injected into the prompt, the content of every contract and rule the task references consolidated into a `spec.md` file in that reviewer's own temporary folder that the prompt directs it to read in full (the linked section when the reference carries a section anchor, the whole file when it does not), and the full list of contract files and rule files by namespace path; the reviewer may consult any file in those lists at its discretion. Each reviewer is instructed to find why the working-tree changes FAIL to satisfy the task spec or to honor the contracts and rules referenced by the task. Verifying that every contract and rule referenced by the task is actually honored in the working-tree changes is an explicit obligation of each reviewer — a referenced contract or rule that is not honored is a FAIL; a contract or rule the reviewer determines should have applied but did not, even if not referenced by the task, is also a FAIL.

   Each reviewer must run every verification it is required to run and every additional check its judgment deems applicable, and must not stop when the first violation is discovered. The four FAIL conditions above and the acceptance-criteria verification protocol are executed in full on every invocation; encountering a violation in one of them does not exempt the reviewer from completing the rest. The goal is that a single review round produces the complete list of fixes the next worker needs to apply, so the next iteration can address every outstanding problem at once instead of one at a time across multiple iterations.

   Each reviewer records its result by producing its own per-reviewer error file. It appends each violation it finds as it discovers it; each violation entry must be independently actionable: precise enough that the next iteration's worker can act on it without having to rediscover the problem from the diff. When a reviewer finds no violation across every verification, it must still create its per-reviewer error file as an empty file as its final act, so that file always exists once that reviewer has run to a verdict. A reviewer's own output is not consulted for the result and has no prescribed format; the result lives entirely in its per-reviewer error file. See [src/commands/.spec/rules/ai/agents.md#the-implement-orchestrator-decides-each-reviewers-verdict-from-its-own-per-reviewer-error-file--deleted-before-inspected-after](/src/commands/.spec/rules/ai/agents.md#the-implement-orchestrator-decides-each-reviewers-verdict-from-its-own-per-reviewer-error-file--deleted-before-inspected-after).

   For each reviewer, after its invocation completes successfully the orchestrator inspects that reviewer's per-reviewer error file:
   - **Absent** — that reviewer did not produce the file it was required to produce, so it did not run to a verdict. The orchestrator launches a fresh invocation of that same reviewer and inspects again, repeating with no maximum count until the file exists; an absent file is never read as a pass and never restarts the worker or consumes an iteration.
   - **Present** — that reviewer has run to a verdict, and its per-reviewer error file is part of the stage's aggregate (below).

   A reviewer invocation that ends in an error rather than a successful completion is governed by the AI runner's retry policy (see [src/ai/.spec/rules/retry.md#the-runner-retries-retryable-errors-and-rate-limits-via-the-tool-interface-events](/src/ai/.spec/rules/retry.md#the-runner-retries-retryable-errors-and-rate-limits-via-the-tool-interface-events)); the per-reviewer inspection above is reached only on a successful completion, so a reviewer that fails before writing cannot be mistaken for a passing review.

   Once the round-completion condition is met (see the weighted-review paragraph above) and the orchestrator has cancelled any reviewers still in a usage-limit wait, it forms the stage verdict on a single linear path, with no per-file presence branching: it reads the per-reviewer error file of every reviewer that ran to a verdict, concatenates their contents in reviewer order with one newline between files, and trims surrounding whitespace and blank lines from the concatenation. See [src/commands/.spec/rules/ai/agents.md#the-review-stage-verdict-is-the-trimmed-concatenation-of-every-per-reviewer-error-file-on-one-linear-path](/src/commands/.spec/rules/ai/agents.md#the-review-stage-verdict-is-the-trimmed-concatenation-of-every-per-reviewer-error-file-on-one-linear-path).
   - When the trimmed aggregate is **empty** — every reviewer passed — proceed to the commit/check stage.
   - When the trimmed aggregate is **non-empty** — at least one reviewer failed — the review failed. The orchestrator writes the aggregate into the `error.log` file inside the temporary folder so it serves as the next iteration's briefing, then restarts the inner loop.

6. **Commit/check stage.** When the review passed — the adversarial review stage found `error.log` empty after trimming — the task is finalized in this order: first, the task's checkbox is flipped from `[ ]` to `[x]` in the plan file and its metrics object is finalized for that task; and when this task is the last open task — so that flipping it leaves no unchecked task in the plan — the plan file is also renamed to mark the plan complete by prepending the marker `V-` to the very start of its filename — ahead of every other part of the name, including the generation-timestamp prefix `/flanders-plan` wrote (see [src/prompts/.spec/rules/ai/skills/plan.md#flanders-plan-prefixes-every-plan-filename-with-the-generation-timestamp](/src/prompts/.spec/rules/ai/skills/plan.md#flanders-plan-prefixes-every-plan-filename-with-the-generation-timestamp)) — so a plan file named `<name>` becomes `V-<name>` and a completed plan's filename begins with exactly one `V-`. Then the orchestrator runs `git add -A` followed by `git commit` with the task's plan number and title verbatim as the commit message (see [.spec/contracts/cli-commands/implement/git-integration.md](/.spec/contracts/cli-commands/implement/git-integration.md)). Because the rename precedes this staging, the commit captures it and the working tree is left clean exactly as after any other accepted task. The inner loop ends with success only after both the plan-file update and the commit have completed.

   When `git commit` exits non-zero, this stage is treated as a failing stage: the checkbox flip is reverted in the plan file, the git output is captured into `error.log`, and the inner loop restarts. The details of this recovery live in [.spec/contracts/cli-commands/implement/git-integration.md](/.spec/contracts/cli-commands/implement/git-integration.md).

## Task metrics persistence
The metrics object of the current task (defined in [.spec/contracts/shared/plan-file-format.md](/.spec/contracts/shared/plan-file-format.md)) is kept up to date in the plan file while the inner loop runs. A user inspecting the plan file at any point during the run finds values that reflect what the tool has consumed on that task so far. The values are written back to the plan file often enough that the file never lags noticeably behind the live UI.

When a task is marked complete (its checkbox flipped to `[x]`), its metrics object reflects the total consumption accumulated over the entire effort on that task, including the cost of any iterations that failed before the one that passed the adversarial review.

## Previous-iteration error briefing
The previous-iteration briefing is a generic addendum appended to the worker prompt automatically whenever `iteration` is greater than 1. It identifies the current iteration number, states that the previous iteration produced a problem to review, and points at the `error.log` file inside the temporary folder. Iteration 1 receives no such addendum.

The decision to add the briefing depends only on the iteration counter — there is no separate flag. Failing stages do not set anything beyond writing to `error.log`; the counter increment in stage 1 of the next iteration is what makes the briefing appear.

`error.log` holds only the most recent failing iteration's context: the build, test, and commit stages overwrite it with their captured output, and the adversarial review stage overwrites it with the aggregated reviewer violations when the review fails — leaving it untouched when the review passes. The reviewers themselves write to their own per-reviewer `error.log` files (each inside its own per-reviewer temporary folder), never to the briefing `error.log`; the orchestrator is what writes the aggregated violations into the briefing `error.log` on a failed review. The same fixed `error.log` file name is used regardless of which stage produced the failure (build, test, review, or commit), so the briefing wording stays generic.

## Hard stop
Exceeding `MAX_ITER` on any single task ends the entire run. The command prints an error that:
- Identifies the task by its line number and title.
- Points at the temporary folder so the user can inspect the per-iteration logs and `error.log`.

Unlike every other exit path, the hard stop intentionally preserves the temporary folder on disk so the user can inspect it; the automatic cleanup defined in [.spec/contracts/cli-commands/implement/workspace.md](/.spec/contracts/cli-commands/implement/workspace.md) is suppressed in this case.

The command then exits non-zero.

## Spec folder immovability
None of the AI agents spawned by the `implement` command — the worker or the reviewer — may write to any `.spec/contracts` folder, any `.spec/rules` folder, any `.spec/flanders` folder, or the `plans/` folder. The `implement` command itself writes to `plans/` to rewrite the checkbox state and the metrics object of an existing task line as work progresses (per the commit/check and task-metrics-persistence sections above and per [.spec/contracts/shared/plan-file-format.md](/.spec/contracts/shared/plan-file-format.md)), and to rename a plan file once, as it finalizes the task that completes the plan, to mark the plan complete (per the commit/check stage above); it does not create new plan files or delete existing ones. See [.spec/contracts/shared/spec-folder-write-authority.md](/.spec/contracts/shared/spec-folder-write-authority.md).
